## Context

garutyunov.com is a Next.js static-export site. `app/page.tsx` (a Server
Component) calls `fetchAllProjectMeta(projects)` from `lib/og.ts` at build time;
`og.ts` fetches each project's live URL with Node `fetch`, parses OG/HTML meta,
and bakes `ResolvedProject[]` into the static page. Today `projects` is a
hand-maintained array in `lib/projects.ts` — the only reason a code change is
needed to add a project.

Each pet project already self-describes on GitHub: the `pet-project` topic
(added by the `pet-project-metadata` conventions) and a `homepage` set to its
published website URL. And every pet project ships a real `og:image` — generated
by the **social-image** skill (workspace#12) — so the live page always has a card
image. So the static array can be replaced by a build-time GitHub query with no
change to the downstream crawl/render.

## Goals / Non-Goals

**Goals:**
- Discover the project list from GitHub at build time (topic `pet-project`,
  owner `gaarutyunov`), including **private** repos whose website is published.
- Reuse `lib/og.ts` unchanged — only swap the data source in front of it.
- **Fail the build loudly** if discovery fails — no fallback, no silent empty
  list (see D5).
- Keep the rendered output identical in shape to today's.

**Non-Goals:**
- Runtime/client-side fetching — discovery stays at build time (static export).
- Changing the crawl/parse logic or the card UI.
- A general multi-owner or org-wide crawler — scoped to `gaarutyunov`.
- Incremental/ISR revalidation — a rebuild refreshes the list (matches current
  static-export model).
- Reading or exposing any project's **source code** — discovery consumes only
  public repo metadata and the published website (see D7).

## Decisions

### D1: REST search `GET /search/repositories?q=topic:pet-project user:gaarutyunov`
One request returns all tagged repos with the fields needed (`name`, `homepage`,
`description`, `created_at`, `fork`, `archived`, `private`). Simpler than listing
all repos and filtering topics client-side, and the result set is tiny.
- *Alternative — `GET /users/gaarutyunov/repos` + filter `topics`:* needs the
  `topics` preview media type per repo or a second call each; more requests.
  Rejected.
- *Alternative — GraphQL:* one typed query, but adds a client/schema for a tiny
  win. Rejected for a plain `fetch` to the REST endpoint (consistent with
  `og.ts`, which uses bare `fetch`).

### D2: Build-time auth via `GITHUB_TOKEN` — **required**
Read `process.env.GITHUB_TOKEN` (GitHub Actions injects it automatically) and
send `Authorization: Bearer`. The token is **required**, not optional: it is what
lets discovery see **private** repos owned by `gaarutyunov` (D7), and it removes
any dependence on the 60 req/hr unauthenticated limit. If the token is absent,
discovery **throws** and the build fails with a clear message — there is no
unauthenticated fallback (consistent with the no-fallbacks stance in D5).

### D3: Map repo → `Project`, homepage → url, GitHub fields → the pipeline fields
`{ id: repo.name, url: repo.homepage, fallback: { name: repo.name,
description: repo.description ?? "", created: repo.created_at.slice(0,10) } }`.
The card image comes from the live page's `og:image` through the existing crawl.
Because every pet project ships an `og:image` generated by the **social-image**
skill (workspace#12) — an invariant of the `pet-project-metadata` conventions —
the crawl always finds a real image. There is **no bundled placeholder icon**: a
missing `og:image` is a project-metadata defect to fix at the source project (run
the social-image skill), not something the site papers over.
- Skip repos with empty `homepage`, and skip `fork`/`archived` repos.

### D4: Exclude the site repo and order deterministically
Filter out `garutyunov.com` by name. Sort by `created` descending (newest first)
for a stable, meaningful order independent of GitHub's API ordering.

### D5: No fallbacks — **fail the build on discovery failure**
`discoverPetProjects()` does **not** swallow errors. On any error, non-OK
response, missing token, or timeout it **throws**, so `next build` fails loudly.
There is no graceful degradation to `[]`, no "last-known" list, and no committed
fallback list. Rationale: a homepage silently shipped with an empty or stale
pet-projects section is worse than a failed build — a failed build is visible in
CI and forces the real problem (outage, bad token, API change) to be fixed before
publishing.

### D6: `lib/projects.ts` keeps the types, loses the data
The `Project` / `ResolvedMeta` / `ResolvedProject` interfaces remain (the
pipeline contract). The exported `projects` array is removed; `app/page.tsx`
calls `discoverPetProjects()` then `fetchAllProjectMeta(...)`. The committed
`public/projects/*.png` icons become dead assets — remove them as cleanup (images
now come from `og:image`).

### D7: Private-source projects with a public website are supported
A pet project's repo may be **private** while its website is published (public
GitHub Pages or a custom domain). Because discovery is authenticated (D2), such a
repo still appears in the search result, and the site uses only its **public
metadata** (`name`, `description`, `created_at`, `homepage`) plus the **published
website** (crawled for `og:*`). The site never reads, links to, or exposes the
repo's source. So a private project can be listed on garutyunov.com without
opening its code — the homepage is the only thing a visitor ever reaches.
- Consequence: discovery does **not** filter by `visibility: public`. Both public
  and private repos carrying the topic (with a homepage) are listed.

## Risks / Trade-offs

- **[GitHub outage/rate-limit or missing token at build → build fails]** →
  intended (D5). The failed build is the signal; fix the outage/token and rebuild.
  No fallback list.
- **[A pet project without `og:image`]** → treated as a metadata defect at the
  source project: run the social-image skill (workspace#12) so the project ships
  an `og:image`. The site does not carry a placeholder for this case.
- **[A repo tagged `pet-project` but not meant to be listed]** → the topic is the
  opt-in; anything tagged (with a homepage) is intended to be listed. Fork/archived
  filter removes obvious noise.
- **[Private repo listed, but source stays closed]** → by design (D7): only public
  metadata + the published site are used; source is never touched.
- **[Description drift between GitHub and the live og:description]** → live
  `og:description` wins (existing pipeline precedence), so the site reflects the
  page, not stale repo text.

## Migration Plan

- Additive then subtractive: introduce `discoverPetProjects()`, switch
  `app/page.tsx` to it, then delete the static `projects` array and the
  `public/projects/*.png` icons in the same PR.
- Rollback: restore the static `projects` array export and point `app/page.tsx`
  back at it — the crawl pipeline is untouched, so rollback is a one-file revert.

## Open Questions

_Resolved from spec review (workspace#11):_
- **Committed fallback list for the outage case?** → **No fallbacks** (D5): fail
  the build instead.
- **Placeholder icon for a missing `og:image`?** → **No placeholder** (D3): every
  project ships an `og:image` via the social-image skill (workspace#12); a missing
  one is fixed at the source project.
- **Filter to `visibility: public`?** → **No** (D7): private repos with a public
  website are supported; discovery is authenticated and uses only public metadata
  + the published site.
