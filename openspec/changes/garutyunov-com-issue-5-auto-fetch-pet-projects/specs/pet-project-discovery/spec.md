## ADDED Requirements

### Requirement: Discover pet projects from GitHub by topic

At build time the site SHALL query GitHub for all repositories owned by
`gaarutyunov` that carry the `pet-project` topic, and use that result as the set
of pet projects to render.

#### Scenario: Repos with the topic are discovered
- **WHEN** the site is built and `gaarutyunov` has repos tagged `pet-project`
- **THEN** each such repo is considered a candidate pet project

#### Scenario: Repos without the topic are ignored
- **WHEN** a `gaarutyunov` repo does not carry the `pet-project` topic
- **THEN** it is not included as a pet project

### Requirement: Use the repo homepage as the project URL

For each discovered repo the site SHALL use the repo's `homepage` field (the
GitHub Pages website URL) as the project URL fed into the metadata crawl.

#### Scenario: Repo has a homepage
- **WHEN** a discovered repo has a non-empty `homepage`
- **THEN** that URL is used as the project's URL for the crawl and as the card link

#### Scenario: Repo has no homepage
- **WHEN** a discovered repo has an empty or missing `homepage`
- **THEN** the repo is skipped (not rendered) rather than linked to a non-site URL

### Requirement: Exclude non-project repos

The discovery SHALL exclude the site's own repository (`garutyunov.com`) even if
it were to carry the topic, so the site never lists itself as a pet project.

#### Scenario: Site repo is filtered out
- **WHEN** the discovery result would contain the `garutyunov.com` repo
- **THEN** it is removed from the pet-project list

### Requirement: Map GitHub metadata to the pipeline fallback

Each discovered repo SHALL be mapped to the pipeline's `Project` shape, with
name/description/created derived from GitHub metadata: name from the repo,
description from the repo description, and `created` from the repo `created_at`.
The card image SHALL come from the live page's `og:image` via the existing crawl
(repos have no icon field). There SHALL be **no bundled placeholder image**:
every pet project is expected to ship an `og:image` generated by the social-image
skill (workspace#12), so a missing image is a source-project defect to fix there,
not something the site substitutes for.

#### Scenario: Name/description/created populated from repo metadata
- **WHEN** a repo is mapped to a project
- **THEN** its name/description/created come from the repo's fields and the crawl
  supplies the card image from the live page's `og:image`

#### Scenario: Live og:image is always present
- **WHEN** a discovered project's live page is crawled
- **THEN** the crawl finds an `og:image` (generated by the social-image skill per
  the metadata conventions), and no placeholder image is used

#### Scenario: Crawl overrides GitHub metadata when live metadata is present
- **WHEN** the project's live page exposes `og:title` / `og:description` /
  `og:image`
- **THEN** those values are used in preference to the GitHub-derived values
  (existing pipeline behavior is preserved)

### Requirement: Deterministic ordering

The rendered pet projects SHALL appear in a stable, deterministic order across
builds (e.g. by creation date) rather than GitHub's incidental API order.

#### Scenario: Stable order
- **WHEN** the site is built twice with the same set of repos
- **THEN** the projects render in the same order both times

### Requirement: Fail the build on discovery failure (no fallback)

Discovery SHALL NOT degrade gracefully: on any GitHub API failure, non-OK
response, rate-limit, timeout, or a missing `GITHUB_TOKEN`, discovery SHALL throw
so that `next build` fails loudly. There SHALL be no fallback to an empty list, a
last-known list, or a committed fallback list.

#### Scenario: GitHub API unreachable at build
- **WHEN** the GitHub discovery request fails, is rate-limited, or times out
  during the build
- **THEN** discovery throws and `next build` fails with a clear error, rather than
  shipping an empty or stale pet-projects section

#### Scenario: Missing token fails the build
- **WHEN** `GITHUB_TOKEN` is not present in the build environment
- **THEN** discovery throws with a clear message and the build fails (there is no
  unauthenticated fallback)

### Requirement: Authenticated discovery includes private repos

Discovery SHALL authenticate with the required `GITHUB_TOKEN` and SHALL include
repositories regardless of visibility — a **private** repo carrying the
`pet-project` topic with a published `homepage` is listed the same as a public
one. The site SHALL consume only the repo's public metadata (name, description,
`created_at`, `homepage`) and the published website; it SHALL NOT read, link to,
or expose the repo's source.

#### Scenario: Private repo with a public website is listed
- **WHEN** a private `gaarutyunov` repo carries the `pet-project` topic and has a
  published `homepage`
- **THEN** it appears in the pet-projects list, sourced from its public metadata
  and its published site, with no access to or exposure of its source code

#### Scenario: Visibility is not filtered
- **WHEN** the discovery result contains both public and private tagged repos
- **THEN** both are listed (discovery does not filter by `visibility: public`)
